* # Архитектура приложения

## Web Layer (REST / UI)

Слой отвечает за HTTP API и взаимодействие с фронтендом или браузером.
На этом уровне располагаются REST-контроллеры, обработка запросов пользователя, выдача страниц и JSON-ответов.

Основные задачи:

* получение запросов на анализ фотографий;
* выдача статистики;
* отображение миниатюр и метаданных;
* экспорт данных.

## Service Layer (Business Logic)

Слой бизнес-логики содержит основную логику приложения.
Именно здесь реализуются сценарии:
* загрузки информации о фотографиях;
* анализа лайков/просмотров;
* расчета динамики;
* агрегации статистики;
* работы с кэшем и миниатюрами.

Сервисный слой изолирует бизнес-логику от деталей хранения данных и внешних API.

## Integration Layer (External Site Integration)

Слой интеграции отвечает за взаимодействие с внешним сайтом фотографий.
Он выполняет:
* HTTP-запросы;
* парсинг HTML;
* извлечение статистики;
* скачивание изображений и миниатюр.

В дальнейшем этот слой можно будет вынести в отдельный сервис или background worker.

## Persistence Layer (Database Access)

Слой доступа к данным работает с PostgreSQL через JPA/Hibernate или JDBC.
Он отвечает за:
* хранение метаданных фотографий;
* сохранение исторической статистики;
хранение информации о скачанных миниатюрах;
* выборки для аналитики и графиков.

Для статистики желательно использовать отдельные таблицы временных рядов (time-series style), чтобы удобно хранить почасовые снимки данных.

## Scheduler / Background Jobs

Фоновый слой отвечает за периодический сбор статистики.
Планируется использование встроенного scheduler-механизма Spring (@Scheduled) для:
* обновления лайков/просмотров;
* проверки новых фотографий;
* скачивания миниатюр;
* очистки устаревших файлов.

На этапе POC этого достаточно без отдельной очереди задач.

## Хранение миниатюр
Локальное файловое хранилище

Для POC миниатюры можно хранить локально на файловой системе внутри отдельной директории, например:

/storage/thumbnails/{photoId}.jpg

В базе данных при этом хранится:
* идентификатор фотографии;
* путь к файлу;
* размер;
* дата загрузки;
* hash файла (опционально).

Такой подход:
* минимально сложный;
* не требует внешних сервисов;
* уменьшает нагрузку на внешний сайт;
* позволяет работать быстрее при отображении галереи.
* Docker deployment

Для Docker желательно использовать volume:

volumes:
- ./storage:/storage

Это позволит:
* не терять миниатюры при перезапуске контейнера;
* отдельно управлять хранилищем;
* упростить резервное копирование.
* Подготовка к Google Cloud

Архитектуру желательно сразу строить так, чтобы локальное файловое хранилище можно было заменить на объектное.

В дальнейшем локальное хранение миниатюр можно заменить на:
* Google Cloud Storage Bucket;
* CDN;
* отдельный image-service.

Для этого лучше сразу сделать абстракцию:
ThumbnailStorageService
* с реализациями:
* LocalThumbnailStorageService
* GcsThumbnailStorageService
* S3ThumbnailStorageService (опционально)

Тогда миграция в cloud не потребует переписывания бизнес-логики.
